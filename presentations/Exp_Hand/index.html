Condition Handling

Advance R Session 1

Author

Diana Garcia

Introduction

Important concepts

  1. Conditions
    • Errors (severe): function should not continue
    • Warnings (mild): something went wrong but it will continue
    • Messages (mildest): inform the user of an action taken
  2. Signals of conditions
    • stop() or rlang::abort(),
    • warning() or rlang::warn(),
    • message()

When to use

  1. Error:
    • We need the process to stop.
    • Message should indicate what is wrong and where to fix the problem.
  2. Warning:
    • Warn of an issue and let the process continue.
    • These can be many per function and will be displayed in aggregate.
    • Message like important change was done or missing info was found.
    • Better not to overuse it.
  3. Messages:
    • Usually when something is done on the behalf of the user
    • E.g. A default value was used or the progress on a long process.
    • Better not to have many messages per function. They are immediately displayed.

Handling conditions

We can override or supplement the default behavior using:

  1. tryCatch()
  2. withCallingHandlres()

tryCatch

It is used for exiting handlers and more suitable for errors.

tryCatch(
  
  error = function(cnd) {
    # code to run when error is thrown
  },
  
  expr = {
    code_to_run_while_handlers_are_active
  }
  
)

When condition is handled it returns to where tryCatch was called.

withCallingHandlers

It is used for calling handlers and more suitable for non-errors conditions.

withCallingHandlers(
  
  warning = function(cnd) {
    # code to run when warning is signalled
  },
  
  message = function(cnd) {
    # code to run when message is signalled
  },
  
  expr = {
    code_to_run_while_handlers_are_active
  }
)

After condition is captured, it returns where the condition was signaled.

Exiting conditions

We could override the default error

f3 <- function(x) {
  tryCatch(
    
    error = function(cnd) 
      NA,
    
    expr = {
      log(x)
    }
    
  )
}

f3("x")
[1] NA

Exiting conditions

tryCatch exits once the condition is signaled:

  • Signal
  • No signal
tryCatch(
  
  message = function(cnd) 
    "There",
  
  expr = {
    message("Here")
    stop("This code is never run!")
  }
)
[1] "There"
tryCatch(
  
  error = function(cnd) 
    10,
  
  expr = {
    message("Hi!")
    1 + 1
  }
)
Hi!
[1] 2

Calling conditions

They help signal a problem and the code execution continues normally.

withCallingHandlers(
  
  message = function(cnd) 
    cat("Caught a message!\n"), 
  
  expr = {
    message("Someone there?")
    message("Why, yes!")
  }
)
Caught a message!
Someone there?
Caught a message!
Why, yes!

Notice: They are applied in order.

PIP Example

Motivation

  • Price Framework data (PFW) is the ‘metadata’ behind each survey in Dataliweb (DLW).
  • Important: for each specific survey we should have only one record (or row/observation).
Objective:
  1. Signal if there are duplicates of PFW per country_code,surveyid_year and survey_acronym.
  2. Log which are the duplicates
  3. And continue with the process using only unique values.

# Function to handle duplicated observations in pfw
unq_pfw <- function(dt,
                    keyVar){

  if(data.table::uniqueN(dt, by = keyVar) != nrow(dt)){

    tryCatch(
      expr = {

        cli::cli_abort()

      },
      error = function(cnd){

        message("There are duplicates in `pfw`")

      }

    dt <- unique(dt, by = keyVar)

Signaling and Handling

  1. Let’s use the condition object
# Function to handle duplicated observations in pfw
unq_pfw <- function(dt,
                    keyVar){

  if(data.table::uniqueN(dt, by = keyVar) != nrow(dt)){

    tryCatch(
      expr = {
        
        dt_d <- dt[duplicated(dt, by = keyVar)]         
        n_rep <- nrow(dt_d)          
        cli::cli_abort(message = "There {?is/are} {n_rep} duplicates in `pfw`",
                       class = c("piperr"),
                       link =  unique(dt_d$link))

      },
      piperr = function(cnd){

        msg <- cat("[", class(cnd)[[1]], "] ",
            cnd$message," for ", cnd$link, "\n", sep = "")
        
        message(msg)

      }
    )

    dt <- unique(dt, by = keyVar)
  }
  return(dt)
}

But what if I don’t want it to be skipped?

Handling

  1. Let’s add another argument
# Function to handle duplicated observations in pfw
unq_pfw <- function(dt,
                    keyVar,                    
                    skip_err = TRUE){

  if(data.table::uniqueN(dt, by = keyVar) != nrow(dt)){

    tryCatch(
      expr = {
        
        dt_d <- dt[duplicated(dt, by = keyVar)]         
        n_rep <- nrow(dt_d)          
        cli::cli_abort(message = "There {?is/are} {n_rep} duplicates in `pfw`",
                       class = c("piperr"),
                       skip = skip_err,
                       link =  unique(dt_d$link),
                       call = sys.call())
        },
      piperr = function(cnd){

      msg <- cat("[", class(cnd)[[1]], "] ",
            cnd$message," for ", cnd$link, "\n", sep = "")
        
        message(msg)
        
      if(!cnd$skip){
        # Abort if you don't want to skip, but after logging
        cli::cli_abort(cnd$message, 
                       call = cnd$call)
      }
        
        } 
    )

    dt <- unique(dt, by = keyVar)
  
    }
  return(dt)
}

My solution

# Function to handle duplicated observations in pfw
unq_pfw <- function(dt,
                    keyVar,                     
                    log_err = TRUE,                     
                    skip_err = TRUE){

    tryCatch(
      expr = {
        
        if(data.table::uniqueN(dt, by = keyVar) != nrow(dt)){
        
        dt_d <- dt[duplicated(dt, by = keyVar)]         
        n_rep <- nrow(dt_d)          
        cli::cli_abort(message = "There {?is/are} {n_rep} duplicates in `pfw`",
                       class = c("dup_pfw", "piperr"),
                       log = log_err,
                       skip = skip_err,
                       link =  unique(dt_d$link),
                       call = sys.call())
        
        }
        
        },
      error = function(cnd){
        
        cli::cli_abort(message = "Untracked error",
                       call = cnd$call)
        
      },
      dup_pfw = function(cnd){
        
      if(cnd$log){ 
        # Log the error
        add_log(cnd)
      }
      
      if(!cnd$skip){
        # Abort if you don't want to skip, but after logging
        cli::cli_abort(cnd$message, 
                       call = cnd$call)
      }
        
        },  
      
      finally = {
        
      dt <- unique(dt, by = keyVar)
      
      } 
    )
    
  return(dt)
}

Other applications

  1. Success and failure values:
    • Return a value for success and another for failure.
  2. Resignal and record
    • More informative error messages and record for later investigation
  3. No default behavior:
    • Using signal() you can create a condition different from a error,warning or message.
